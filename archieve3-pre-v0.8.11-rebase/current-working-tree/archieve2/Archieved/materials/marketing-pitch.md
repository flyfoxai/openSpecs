# 停！别再让 AI 直接给你写代码了 —— 谈谈 AI 编程时代的“架构失控”

你是不是也经历过这样的绝望时刻：
给了 AI 一段需求，它吭哧吭哧几秒钟吐出几百行代码。一开始看着挺像回事，结果一跑报错。你把报错丢给它，它又改了另外几百行。三个回合下来，你的业务逻辑已经被改得面目全非，原本清晰的架构变成了意大利面条，最后你不得不含泪 Git Revert。

**痛点根本不在于模型不够聪明，而在于我们把“推理”和“编码”混为一谈了。**

人类高级程序员拿到复杂需求，一定会先画架构图、设计表结构、确认流转状态，然后才写代码。凭什么我们要求 AI 一步登天？

为了终结这种“Prompt -> 乱写代码 -> 报错 -> 再次 Prompt”的死循环，我们开源了 **OpenSpecs (Speckit Layered)**。

### 🚀 什么是 OpenSpecs？
这是一个专为 AI Agent 设计的**文档驱动开发框架（Document-Stage Framework）**。
它相当于给狂奔的 AI 套上了一个“工程化护栏”。

### 💡 它到底有多硬核？

**1. 强制的“思维链落盘” (Chain of Thought in Files)**
AI 不允许直接改代码。它必须被强制按照我们的两层工作流执行：
先执行 `sp.specify` (需求定义) -> `sp.clarify` (消除歧义) -> `sp.gate` (状态门禁)
最后才进入 `sp.plan` (计划制定)。
AI 每一步的推理都变成 Markdown 文件固化在本地。你能看懂，你能 Review 并在必要时介入修改。

**2. 解决“上下文爆炸”的 Query-First Memory 路由引擎**
别再傻乎乎地把几十张表、几十个接口的代码库一股脑丢给 1M Context 的大模型了，它会严重注意力失焦！
OpenSpecs 引入了强约定的 `.specify/memory` 路由表。Agent 进来后第一步必须看索引表，第二步精准提取当前 Feature 的“最小读集”。Token 消耗大幅降低，逻辑分析准度史诗级提升。

**3. 解决“项目过大”的 Workset 局部工作面**
当需求庞大时，框架会自动进行任务切块（Workset）。AI 每次干活只背负当前闭环区域的上下文，就像微服务一样解耦了思维包袱。

**4. 全平台无缝兼容的协议层**
底层规范核心始终是 `sp.*`。无论你是用 Claude Code 敲 `/sp.specify`，还是用 Codex 敲 `/prompts:sp.specify`，它都能完美自动适配。
