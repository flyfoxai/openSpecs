# `sp` 自动开发最小补充规范

> 目标：在不新增命令、不进入 `implement` 阶段的前提下，定义当前 `sp` 文档体系还需要补哪些最小规格，才能更稳定地支撑后续自动开发。

## 1. 结论

当前 `sp` 文档体系已经足以作为：

- 中型项目的业务澄清输入
- 中型项目的交付设计输入
- 多 agent 协同拆解任务的文档基础
- 分阶段、分模块的半自动开发输入

当前 `sp` 文档体系还不足以直接作为：

- 单轮低人工介入的整项目代码生成输入
- `50` 个界面、`50` 条流程、`100` 张表的一次性稳定自动实现输入

问题不在分层方向，而在“规格精度”。

现在这套文档更像：

- 交付设计规格

而还不是：

- 可直接执行的一体化机器规格

## 2. 适用判断

如果目标是下面这些场景，当前体系已经够用：

- 先把业务和边界讲清
- 让不同 agent 产出一致的分析与设计文档
- 把大 feature 拆成模块、页面、接口、表和验收项
- 为后续实现阶段准备高质量前置输入

如果目标是下面这些场景，当前体系还不够：

- 让 agent 直接稳定生成完整前后端代码
- 让 agent 在大规模项目中自主判断所有改动范围
- 让 agent 无需人工补全就生成精确数据库迁移、接口实现、页面状态逻辑和自动化测试

## 3. 当前体系已经具备的能力

结合 [sp-auto-development-readiness.md](sp-auto-development-readiness.md) 与现有模板，当前体系已经提供：

- 第一层的目标、范围、角色、规则、流程、状态、页面与异常
- 第二层的交付范围、页面映射、用例矩阵、领域模型、表级目录、API 契约、权限矩阵、副作用、非功能约束、任务与验收
- 稳定编号基础：
  - `ROLE-*`
  - `OBJ-*`
  - `RULE-*`
  - `STATE-*`
  - `FLOW-*`
  - `SCREEN-*`
  - `UC-*`
  - `MOD-*`
  - `API-*`
  - `TABLE-*`
  - `ACC-*`

这已经足以支撑：

- 文档阶段闭环
- 中型 feature 的设计拆解
- 人工实现或半自动实现前的交接

## 4. 当前还差的最小补充

若希望更稳定地支撑后续自动开发，至少还要把下面六类信息写得更结构化。

### 4.1 UI 交互精度

当前 `ui/screen-*.md` 已经能表达页面目标和业务动作，但还应补强：

- 页面状态：
  - 初始态
  - 加载态
  - 空态
  - 错误态
  - 无权限态
- 区块级动作与显隐条件
- 提交前校验与提交后反馈
- 字段与接口字段的绑定关系
- 页面动作对应的 `UC-*` 与 `API-*`

否则 agent 能画出页面职责，但还不够稳定地产出页面实现。

### 4.2 API 契约精度

当前 `delivery/07-api-contracts.md` 已经定义了用途、方法、路径、请求体、响应体、错误码、幂等要求，但还应补强：

- 字段类型
- 必填与可空
- 枚举值
- 默认值
- 分页与排序约束
- 权限前提
- 状态前提
- 副作用声明

否则 agent 能列接口，但难以稳定实现接口代码与测试。

### 4.3 数据库规格精度

当前 `delivery/tables/table-*.md` 已经定义了字段、约束、索引、外键、读写来源和归档要求，但还应补强：

- 字段类型与长度
- 默认值
- 是否允许空值
- 更新策略
- 软删除还是硬删除
- 审计字段触发规则
- 状态字段与状态机的精确对应

否则 agent 能建出表清单，但难以稳定生成迁移与仓储层实现。

### 4.4 权限与数据范围精度

当前 `delivery/08-permissions-matrix.md` 已经覆盖角色、动作、资源范围和条件，但还应补强：

- 页面可见性
- 按钮可见性
- 字段可编辑性
- 数据行级范围
- 跨组织访问限制
- 审批代理与代办权限

否则 agent 能知道“谁能做什么”，但还不能稳定写出页面和接口里的权限判断。

### 4.5 事件与副作用精度

当前 `delivery/09-events-and-side-effects.md` 已经覆盖触发源、触发条件、下游动作和失败处理，但还应补强：

- 幂等键
- 重试策略
- 顺序要求
- 补偿策略
- 通知模板来源
- 审计日志内容

否则 agent 能知道有副作用，但难以稳定落异步任务、消息和审计链路。

### 4.6 验收与追踪精度

当前 `delivery/12-test-and-acceptance.md` 已经覆盖测试目标和验证方式，但还应补强：

- 每个验收项对应的前置数据
- 操作步骤
- 预期结果
- 异常结果
- 覆盖的 `SCREEN-* / API-* / TABLE-* / RULE-*`

否则 agent 能知道要测什么，但还不能稳定生成端到端测试和接口测试。

## 5. 考勤申请冒烟复核

下面用“员工请假申请”这条链路，检验当前文档是否足够支撑自动开发。

### 5.1 当前已经能表达的链路

当前体系已经能稳定写出：

- `ROLE-EMPLOYEE` 发起请假
- `ROLE-MANAGER` 审批请假
- `FLOW-LEAVE-APPLY`
- `STATE-DRAFT / STATE-PENDING / STATE-APPROVED / STATE-REJECTED`
- `SCREEN-LEAVE-APPLY`
- `SCREEN-LEAVE-APPROVAL`
- `UC-LEAVE-SUBMIT`
- `API-LEAVE-CREATE`
- `API-LEAVE-APPROVE`
- `TABLE-LEAVE_REQUEST`
- `TABLE-LEAVE_APPROVAL`
- `ACC-LEAVE-APPROVE-SUCCESS`

这条链路说明：

- 当前体系已经能把业务、页面、接口、表、验收串起来

### 5.2 当前还会卡住的地方

如果直接让 agent 写代码，还会卡在这些问题：

- 申请页哪些字段在不同假种下显示或隐藏
- 假期余额是在提交前校验还是提交后异步锁定
- 审批通过后通知是同步发还是异步发
- 审批驳回是否回滚余额占用
- 请假单撤回后哪些状态允许再次编辑
- 审批接口的幂等键如何定义
- 详情页中谁可以看到审批意见、谁只能看到结果
- 列表页默认排序、筛选和分页规则是什么
- 表字段里时区、日期边界和半天请假的存储方式是什么

这些问题都不是“有没有文档”的问题，而是“文档有没有精确到可执行实现”的问题。

## 6. 中型项目的可支撑性判断

针对 `50` 个界面、`50` 条流程、`100` 张表的中型项目，当前体系的判断如下：

### 6.1 可以支撑的范围

- 业务域拆解
- 需求澄清
- 交付设计
- 模块划分
- 数据对象梳理
- 接口清单管理
- 表级规格管理
- 任务与验收拆解

### 6.2 不能直接保证的范围

- 一次性完整生成前后端实现
- 无人工校对地生成所有迁移与测试
- 大规模跨模块自动修改时的稳定范围控制

### 6.3 关键前提

只有在下面这些前提都成立时，当前体系才适合作为后续自动开发输入：

- 严格使用稳定编号
- 按业务域拆 flow
- 按模块拆 ui
- 按主题拆 API 契约
- 按表逐表成文
- 每个任务必须回链页面、接口、表或验收项
- `analysis.md` 必须显式指出是否达到自动开发最低标准

## 7. 不新增命令时的补强办法

本仓库已经决定不新增新命令，因此补强方式应放在现有命令输出上。

建议这样做：

- `sp.ui`
  - 把页面文档写到“区块级动作 + 状态级行为 + 接口绑定”
- `sp.plan`
  - 把第二层文档写到“可回链、可拆分、可机读”
- `sp.tasks`
  - 把任务直接绑定 `SCREEN-* / API-* / TABLE-* / ACC-*`
- `sp.analyze`
  - 明确判断当前 feature 是否达到“自动开发最低通过线”
  - 若 feature 已达到中型项目规模，还应执行一次面向抽样和关键链路的冒烟测试

这意味着：

- 不需要新命令
- 但需要把现有产物写得更硬、更细、更结构化

## 8. 最终判断

当前这套 `sp` 文档体系，对中型项目来说已经不是“想法”，而是可执行的文档框架。

但如果目标是后期全自动开发，它现在还差最后一层约束：

- UI 交互精度
- API schema 精度
- 数据字段精度
- 权限范围精度
- 事件幂等与补偿精度
- 验收步骤精度

因此最准确的结论是：

- 当前体系已经足够支撑中型项目文档开发
- 当前体系已经足够支撑分阶段自动开发
- 当前体系还不足以支撑一把梭的全自动代码开发
- 若补齐本页列出的最小补充规范，就可以更稳地进入后续自动开发阶段

对应的执行式检查见：

- [docs/sp-medium-project-smoke-test.md](sp-medium-project-smoke-test.md)
