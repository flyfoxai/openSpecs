# `sp` 自动开发通过线清单

> 目标：定义一个 feature 在文档阶段必须达到什么标准，才允许作为后续自动实现的输入。

## 1. 结论

一个 feature 只有同时通过：

- 第一层业务澄清通过线
- 第二层交付设计通过线
- 跨文档追踪通过线
- 自动开发精度通过线

才可以进入后续自动实现阶段。

这里的“进入”不是指已经可以一键生成整套系统，而是指：

- 可以作为后续模块化自动开发输入
- 可以作为多 agent 协同自动开发输入
- 可以作为实现阶段的稳定基线

如果任一关键项不通过，正确动作不是硬进实现，而是回退到对应 `sp` 步骤补文档。

## 2. 适用范围

本清单适用于：

- 单个 feature 的文档阶段验收
- 中型项目中的模块级自动开发前检查
- `sp.analyze` 给出最终结论时的统一判断标准

本清单不用于判断：

- 整个项目是否已经全部可上线
- 某个技术框架是否已经选定
- 生产代码是否已经可运行

## 3. 总体判定规则

判定分为三档：

- `PASS`：可作为后续自动实现输入
- `PASS WITH RISKS`：可进入受控自动实现，但必须先显式记录风险
- `BLOCKED`：不得进入后续自动实现

出现以下任一情况，默认 `BLOCKED`：

- 关键页面没有稳定 `SCREEN-*`
- 关键用例没有稳定 `UC-*`
- 关键接口没有稳定 `API-*`
- 关键表没有稳定 `TABLE-*`
- 页面、用例、接口、表、验收之间无法回链
- 存在未决业务规则却被当成既定结论
- 权限边界、副作用或验收条件仍然缺失

## 4. 第一层通过线

第一层至少必须满足：

- `spec.md` 已明确目标、范围、角色、成功标准、非目标
- `clarifications.md` 已明确规则、默认值、覆盖关系、异常、边界条件
- `flows/*` 已覆盖主流程、关键交互时序、状态流转
- `ui/screen-map.md` 已形成稳定页面清单
- `ui/screen-*.md` 已覆盖关键页面
- `gate.md` 已给出 `PASS` 或 `PASS WITH OPEN QUESTIONS`
- `bundle.md` 已能独立说明业务主线

第一层若要继续作为自动实现输入，还必须补足：

- 关键角色、对象、规则、状态、流程、页面具备稳定编号
- 开放问题与稳定规则分离
- 主流程闭环，关键分支可见
- 页面能够回链到流程与规则

## 5. 第二层通过线

第二层至少必须满足：

- `plan.md` 已明确范围、阶段、模块边界、验收入口
- `delivery/02-screen-to-delivery-map.md` 已建立页面到交付对象的映射
- `delivery/03-use-case-matrix.md` 已建立页面动作到用例的映射
- `delivery/05-data-entity-catalog.md` 已建立对象字典
- `delivery/06-table-index.md` 与 `delivery/tables/table-*.md` 已建立逐表规格
- `delivery/07-api-contracts.md` 已建立接口契约
- `delivery/08-permissions-matrix.md` 已建立权限矩阵
- `delivery/09-events-and-side-effects.md` 已建立副作用说明
- `delivery/12-test-and-acceptance.md` 已建立验收入口
- `tasks.md` 已建立任务拆解与追踪对象
- `analysis.md` 已给出最终一致性判断

第二层若要继续作为自动实现输入，还必须满足：

- `plan.md` 已给出自动开发就绪结论
- `tasks.md` 每项任务都有主要追踪对象
- `analysis.md` 已明确当前 feature 是否达到自动开发最低通过线
- 若 feature 已达中型项目规模，`analysis.md` 已记录冒烟范围、抽样范围和抽样结论

## 6. 追踪通过线

一个 feature 至少要建立下面这些关键回链：

- `SCREEN -> FLOW -> RULE`
- `SCREEN -> UC`
- `UC -> API`
- `API -> OBJ -> TABLE`
- `TASK -> SCREEN / API / TABLE / ACC`
- `ACC -> SCREEN / API / TABLE / RULE`

如果这些回链只能靠自然语言猜，而不是文档显式写出，则视为未通过。

## 7. 精度通过线

### 7.1 UI 精度

关键页面必须写清：

- 页面状态：初始、加载、空、错误、无权限
- 区块显隐条件
- 字段可编辑条件
- 提交前校验
- 提交后反馈
- 关键动作绑定的 `UC-*`
- 必要时绑定 `API-*`

### 7.2 API 精度

关键接口必须写清：

- 方法与路径
- 请求字段
- 响应字段
- 字段类型
- 必填、可空、默认值、枚举
- 错误码
- 权限前提
- 状态前提
- 幂等要求
- 副作用

### 7.3 数据精度

关键表必须写清：

- 字段名
- 类型与长度
- 是否必填
- 是否可空
- 默认值
- 主键与唯一约束
- 索引
- 外键与关联
- 删除策略
- 审计字段
- 状态字段映射

### 7.4 权限精度

关键权限必须写清：

- 角色
- 动作
- 资源范围
- 条件
- 是否允许
- 页面显隐
- 字段可编辑
- 数据范围限制

### 7.5 副作用精度

关键副作用必须写清：

- 触发源
- 触发条件
- 下游动作
- 同步或异步
- 幂等键
- 重试或补偿
- 失败处理

### 7.6 验收精度

关键验收项必须写清：

- 前置数据
- 测试目标
- 验证方式
- 来源规则
- 来源页面
- 来源接口
- 来源表

## 8. 中型项目专项要求

当项目规模接近：

- `50` 个界面
- `50` 条流程
- `100` 张表

还必须额外满足：

- flow 按业务域拆分
- ui 按模块拆分
- API 契约按主题拆分
- 表规格按表单独成文
- 索引文件能维护编号、名称和关系
- `plan.md` 只做总索引，不承载全部细节

如果没有这些拆分规则，即使单个文档写得很细，也不应视为通过。

## 9. 考勤申请冒烟判定

以“考勤请假申请”为例，一个可通过的 feature 至少要做到：

- `SCREEN-LEAVE-APPLY` 已写明不同假种下的字段显隐与可编辑条件
- `UC-LEAVE-SUBMIT` 已回链到 `API-LEAVE-CREATE`
- `API-LEAVE-CREATE` 已写明字段类型、必填项、错误码、幂等要求
- `TABLE-LEAVE_REQUEST` 已写明日期字段、状态字段、删除策略、审计字段
- `ROLE-MANAGER` 在审批页的页面显隐和字段权限已明确
- 审批通过后的通知、余额扣减、审计日志已写入副作用文档
- `ACC-LEAVE-APPROVE-SUCCESS` 已写明前置数据和验证方式

如果上面这些只写了一半，例如已经有页面和接口名，但没有字段约束、幂等键、显隐条件和验收前置数据，则仍然只能判为：

- `PASS WITH RISKS`

如果连 `SCREEN -> UC -> API -> TABLE -> ACC` 都串不起来，则应直接判为：

- `BLOCKED`

## 10. `sp.analyze` 最终输出建议

`analysis.md` 在最终结论处建议固定输出：

- 总结论：`PASS` / `PASS WITH RISKS` / `BLOCKED`
- 通过项
- 风险项
- 阻塞项
- 缺口分类：
  - UI 精度
  - API 精度
  - 数据精度
  - 权限精度
  - 副作用精度
  - 验收精度
- 建议回退步骤

若 feature 已接近中型项目规模，建议同时附上：

- 冒烟范围
- 抽样统计
- 关键对象是否纳入样本
- 冒烟结果对最终 verdict 的影响

## 11. 最终判断

这份通过线的意义不是把文档做得更厚，而是把“哪些信息已经足够进入后续自动实现”判断得更硬。

最准确的使用方式是：

- 用 `sp.plan` 形成第二层完整交付包
- 用 `sp.tasks` 建立任务与对象追踪
- 用 `sp.analyze` 对照本清单做 feature 级最终判定

只有这样，文档阶段才真正能成为后续自动实现的稳定输入。

中型项目的具体检查动作见：

- [docs/sp-medium-project-smoke-test.md](sp-medium-project-smoke-test.md)
