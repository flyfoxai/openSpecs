# `sp` 文档体系的自动开发就绪度

> 目标：评估当前 `sp` 文档体系是否足以支撑中型项目的后续自动开发，并给出补强标准。

## 1. 评估结论

当前文档体系已经足以支持：

- 中型项目的业务澄清
- 中型项目的交付设计
- 人工开发或半自动开发前的高质量文档准备

当前文档体系还不足以单独支持：

- 低人工介入的全自动代码生成
- 面向 `50` 个界面、`50` 条流程、`100` 张数据库表的稳定代码落地

核心原因不是方向错误，而是规格粒度仍然偏粗。

当前体系已经覆盖：

- 目标、范围、角色、规则、状态、边界
- 主流程、分支流程、异常流程
- 页面清单、页面卡片、表单型原型
- 交付范围、模块边界、任务拆解、验收映射

但当前体系仍缺少稳定的机器可执行规格层，包括：

- 全局编号与追踪规则
- 页面级到接口级的精确回链
- 数据库表级规格
- API 请求与响应契约
- 权限矩阵
- 副作用与异步事件说明
- 非功能约束
- 大规模拆分规则

## 2. 冒烟测试方法

本页采用“考勤申请”作为真实业务冒烟样本。

样本特征：

- 多角色：员工、主管、HR、系统管理员
- 多流程：请假、补卡、加班、出差、撤回、驳回、转审
- 多状态：草稿、待提交、待审批、已通过、已驳回、已撤回
- 多页面：申请页、审批页、列表页、详情页、规则页、配置页
- 多数据对象：申请单、审批记录、假期余额、组织关系、通知记录

评估标准：

1. 文档是否能讲清业务
2. 文档是否能支撑多人协作开发
3. 文档是否能被 agent 直接转换成数据库、接口、页面和测试实现

## 3. 冒烟测试结果

### 3.1 已经满足的部分

用当前第一层文档，可以比较稳定地表达：

- 角色与职责
- 业务对象
- 主线流程、分支和异常
- 业务规则与优先级
- 状态流转
- 页面清单与页面卡片
- 表单型页面的数据骨架

这意味着当前体系在“讲清业务”和“形成交付输入”两个目标上是成立的。

### 3.2 仍然不满足的部分

如果要让 agent 继续自动产出实现，当前文档仍然会卡在下面这些地方：

- 不知道每个页面对应哪些接口
- 不知道每个接口对应哪些数据库表和字段
- 不知道字段类型、约束、索引、主外键和唯一性
- 不知道审批通过后有哪些副作用
- 不知道权限边界如何精确落地
- 不知道并发、幂等、审计、时区、保留策略如何处理
- 不知道当界面数量和流程数量增大时，文档该如何分卷

## 4. 采用的补强结论

为了支撑中型项目的自动开发，`sp` 仍然保持两层，但第二层必须扩大为“交付设计 + 机器可执行规格”。

不新增命令，只增强现有产物。

增强后的第二层至少要补这些规格：

- `screen-to-delivery-map`
- `use-case-matrix`
- `domain-model`
- `entity-catalog`
- `table-index`
- `table-spec`
- `api-contracts`
- `permissions-matrix`
- `events-and-side-effects`
- `non-functional-requirements`
- `test-and-acceptance`

## 5. 自动开发最低通过标准

若一个 feature 需要作为后续自动开发输入，至少应满足：

- 每个角色有稳定 `ROLE-*` 编号
- 每个业务对象有稳定 `OBJ-*` 编号
- 每条规则有稳定 `RULE-*` 编号
- 每个状态有稳定 `STATE-*` 编号
- 每条流程有稳定 `FLOW-*` 编号
- 每个界面有稳定 `SCREEN-*` 编号
- 每个用例有稳定 `UC-*` 编号
- 每个模块有稳定 `MOD-*` 编号
- 每个接口有稳定 `API-*` 编号
- 每张表有稳定 `TABLE-*` 编号
- 每个验收项有稳定 `ACC-*` 编号

并且必须能建立以下回链：

- `SCREEN -> FLOW -> RULE`
- `SCREEN -> UC -> API`
- `API -> OBJ -> TABLE`
- `TASK -> MOD / API / TABLE / SCREEN`
- `ACC -> SCREEN / API / TABLE / RULE`

## 6. 中型项目规模化要求

当项目达到 `50` 个界面、`50` 条流程、`100` 张表时，不应继续沿用单文件堆叠方式。

应采用：

- 按业务域拆分流程目录
- 按模块拆分页面目录
- 按主题拆分接口契约
- 按表一表一文档的方式拆分数据库规格
- 用索引文件维护编号、名称和依赖关系

如果没有这些拆分规则，自动开发阶段会出现：

- 上下文窗口过大
- 命名漂移
- 回链断裂
- 重复生成
- agent 无法稳定判断修改范围

## 7. 对当前仓库的具体结论

当前仓库原有文档体系是合格的第一版。

但若目标是“后续全自动代码开发”，则必须补强为：

- 有稳定编号体系
- 有目录级拆分规则
- 有数据库与接口规格文档
- 有权限与副作用规格文档
- 有非功能约束文档
- 有跨文档追踪规则
