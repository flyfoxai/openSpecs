# 分层业务逻辑框架

> 目标：先把业务讲清楚，再把业务变成交付设计文档，避免需求直接跳到生产代码。

## 1. 总体原则

整个项目只采用两层推进：

1. 第一层：业务澄清层
2. 第二层：交付设计层

这两层不是按文档类型区分，而是按问题类型区分：

- 第一层只回答业务问题
- 第二层只回答交付问题

默认表达方式固定为：

- `Markdown`：业务说明、规则、页面卡片、交付说明
- `Mermaid`：主流程、时序、状态
- `JSON Forms`：配置型页面和表单型页面原型

## 1.1 面向自动开发的完成标准

如果目标只是支持人工开发，这两层已经足够。

如果目标是支持后续自动开发，这两层仍然成立，但必须额外满足：

- 第一层产物要有稳定编号与可追踪关系
- 第二层不仅要有交付设计，还要有机器可执行规格
- 页面、流程、规则、接口、表、任务、验收之间必须可回链
- 当规模达到 `50` 个界面、`50` 条流程、`100` 张表时，文档必须支持按域拆分

## 1.2 全局编号与追踪规则

两层文档统一采用稳定编号。

建议至少使用以下编号前缀：

- `ROLE-*`
- `OBJ-*`
- `RULE-*`
- `STATE-*`
- `STAGE-*`
- `FLOW-*`
- `STEP-*`
- `DEC-*`
- `EX-*`
- `SCREEN-*`
- `SECTION-*`
- `ACTION-*`
- `FIELD-*`
- `UC-*`
- `MOD-*`
- `API-*`
- `TABLE-*`
- `ACC-*`

这些编号的作用：

- 让 agent 能稳定回链而不是靠自然语言猜测
- 让大规模文档可以拆分后再重新组装
- 让任务、测试、页面、接口和数据对象之间可以建立追踪矩阵

### 1.2.1 ID 形态规则

ID 的目标不是“好看”，而是让大模型和人都能一眼识别。

统一规则：

- 全部使用大写前缀和连字符，例如 `FLOW-LEAVE-001`
- 前缀必须直接表达对象类型，避免一个前缀承载多种含义
- 中间段必须表达业务域、feature 或主题归属，例如 `LEAVE`、`ATTENDANCE`
- 末段优先使用稳定语义段或定宽序号，不混用随意写法
- 同一类对象在同一 feature 内只允许一种主写法
- ID 本身只承担“识别”和“粗归属”，不要把整条父子路径都塞进 ID

推荐分成两类：

- 语义型锚点 ID：适合角色、状态、少量长期稳定对象，例如 `ROLE-EMPLOYEE`、`STATE-REQUEST-DRAFT`
- 序号型结构 ID：适合流程步骤、判断点、页面区块、动作、字段，例如 `FLOW-LEAVE-001`、`STEP-LEAVE-010`、`ACTION-LEAVE-030`

### 1.2.2 归属识别与自然语言映射

为了让模型在有限上下文里快速完成“自然语言 -> 精确对象”的映射，凡是需要被引用、修改或回链的对象，都应尽量同时提供以下字段：

- `ID`
- `Canonical Name`
- `Alias Keywords`
- `Owner ID`
- `Source Rules / Source Flows / Source Screens`

设计要求：

- `Canonical Name` 只保留一个主名称，避免正文里同物多名
- `Alias Keywords` 收纳常见口语、业务别称、历史称呼，例如“提交按钮 / 送审按钮 / 确认提交”
- `Owner ID` 只写直接归属，例如 `ACTION-*` 归属于某个 `SCREEN-*` 或 `SECTION-*`
- flow 和 ui 中的低层对象必须能从自然语言描述反查回唯一 ID，若不能唯一反查，就说明编号或别名设计还不够好

因此，对 flow 和 ui 来说，稳定编号不是只给 `FLOW-*` 和 `SCREEN-*` 就够了，还应继续下钻到：

- flow：`STEP-*`、`DEC-*`、`EX-*`
- ui：`SECTION-*`、`ACTION-*`、`FIELD-*`

## 2. 全局推进链路

每个需求都按同一条链路推进：

1. 输入原始需求
2. 第一层澄清业务目标、角色、规则、状态和边界
3. 第一层通过澄清闸门
4. 第二层把稳定业务结果转成交付设计文档
5. 第二层通过交付闸门
6. 文档阶段结束，并输出可交接产物

## 3. 第一层：业务澄清层

### 3.1 目标

第一层要解决四件事：

1. 这个需求到底要解决什么业务问题
2. 谁参与、谁决策、谁执行、谁受影响
3. 正常流程、分支流程、异常流程分别是什么
4. 规则、状态、边界和未决问题分别是什么

### 3.2 输入

第一层输入通常是非结构化材料：

- 一段需求描述
- 会议纪要
- 聊天记录
- 现有产品行为
- 用户举例

### 3.3 核心模块

第一层建议固定拆成八个模块：

1. 业务目标
2. 参与角色
3. 核心对象
4. 主线流程
5. 分支与异常
6. 业务规则与决策表
7. 状态与边界条件
8. 待确认问题与验收样例

### 3.3.1 第一层内部推进顺序

第一层虽然统称“业务澄清层”，但内部仍应保持由宏观到微观的顺序：

1. 需求目标与范围
2. 业务逻辑与规则
3. 业务框架
4. 业务流程
5. 页面与交互表达

其中“业务框架”不是新增顶层层级，而是第一层内部必须显式写出的中间产物。

它至少要回答：

- 业务主线由哪几个阶段组成
- 每个阶段由谁主导、谁参与、谁受约束
- 核心对象沿主线如何流转
- 哪些判断点决定流程分叉
- 哪些能力在当前 feature 内，哪些明确留给后续扩展

为了保证不同 feature 写法一致，业务框架建议固定使用以下顺序：

1. `Mainline Stages`
2. `Stage Responsibility Boundaries`
3. `Object Flow Backbone`
4. `Top-Level Decision Points`
5. `Capability Boundaries`

这五项不要求写成长文，但不应缺项，也不建议调换顺序。

如果没有先把业务框架写出来，就直接进入流程图和页面卡片，后续很容易出现：

- 流程图画得很完整，但主线阶段划分不稳定
- 页面很多，但页面责任边界不稳定
- 第二层能写接口和表，却仍然要回头猜业务骨架

### 3.4 标准产物

第一层完成后，至少应产出以下内容：

- `01-business-summary.md`
- `02-actors-and-objects.md`
- `03-rules-and-decisions.md`
- `03-business-framework.md`
- `03-traceability-ids.md`
- `04-flows/main-flow.mmd`
- `04-flows/sequence.mmd`
- `04-flows/state.mmd`
- `04-flows/index.md`
- `05-screens/screen-map.md`
- `05-screens/screen-xx.md`
- `05-screens/index.md`
- `06-jsonforms/schema.json`
- `06-jsonforms/uischema.json`
- `06-jsonforms/data.example.json`
- `open-questions.md`
- `acceptance-examples.md`

### 3.5 产物标准

第一层产物必须满足：

- 可复核：别人能看懂，不靠口头补充
- 可追问：每条规则都能继续细化
- 可判断：规则和状态能判定对错
- 可交接：第二层能直接接手，不重新猜业务
- 可追踪：角色、规则、状态、流程、页面都有稳定编号
- 可分层：先看到业务骨架，再看到流程与页面细节

### 3.6 第一层闸门

满足以下条件才允许进入第二层：

- 主流程闭环
- 关键分支齐全
- 主要异常已覆盖
- 规则优先级明确
- 状态流转明确
- 页面清单明确
- 阻塞性交付问题已显式暴露

## 4. 第二层：交付设计层

### 4.1 目标

第二层不再讨论“用户到底想要什么”，而是把第一层结果转成可以复核、可以交接、可以独立结束的交付设计文档。

### 4.2 输入

第二层输入不是原始需求，而是第一层已经稳定的业务产物包。

第二层读取第一层时，建议按以下顺序消费：

1. `spec.md` 中的目标、范围、成功标准
2. `clarifications.md` 中的规则、状态、边界与业务框架
3. `flows/*` 中的主线、时序与状态
4. `ui/*` 中的页面责任与动作
5. `bundle.md` 中的业务框架快照、稳定项、风险项与交接约束

### 4.3 核心模块

第二层建议固定拆成十个模块：

1. 交付范围与 PRD 收敛
2. 页面与交互映射
3. 用例矩阵与页面责任
4. 领域模型与数据结构
5. 数据库表与字段规格
6. 模块边界与接口契约
7. 权限矩阵与副作用约束
8. 非功能约束
9. 交付任务拆解
10. 测试、验收与一致性约束

### 4.4 标准产物

第二层完成后，至少应产出以下内容：

- `plan.md`
- `delivery/01-prd.md`
- `delivery/02-screen-to-delivery-map.md`
- `delivery/03-use-case-matrix.md`
- `delivery/04-domain-model.md`
- `delivery/05-data-entity-catalog.md`
- `delivery/06-table-index.md`
- `delivery/tables/table-*.md`
- `delivery/07-api-contracts.md`
- `delivery/08-permissions-matrix.md`
- `delivery/09-events-and-side-effects.md`
- `delivery/10-non-functional-requirements.md`
- `delivery/11-module-boundaries.md`
- `delivery/12-test-and-acceptance.md`
- `tasks.md`
- `analysis.md`

### 4.5 第二层重点动作

第二层要完成的转换包括：

- 把业务页面清单变成页面交付清单
- 把页面和流程变成可执行用例矩阵
- 把业务规则变成接口规则、校验规则和数据契约
- 把核心对象变成实体目录、表目录和字段规格
- 把状态机变成文档级状态流转约束
- 把审批、通知、审计之类联动变成副作用约束
- 把权限差异变成可执行权限矩阵
- 把验收样例变成测试样例
- 把待确认问题转成风险项或阻塞项

如果目标是支撑后续自动开发，第二层还必须把以下精度显式写进文档：

- UI 精度：页面状态、区块显隐条件、字段可编辑条件、提交反馈、`UC-*` 与 `API-*` 绑定
- API 精度：请求字段、响应字段、错误码、分页过滤排序约束、幂等要求、权限前提
- 数据精度：字段类型、长度、必填、可空、默认值、唯一约束、删除策略、状态字段映射
- 权限精度：角色、动作、资源范围、条件、页面显隐、字段可编辑
- 副作用精度：触发条件、幂等键、同步异步、重试或补偿、失败处理
- 验收精度：前置数据、来源规则、来源页面、来源接口、来源表、验证方式

### 4.6 第二层闸门

满足以下条件才允许结束文档阶段：

- 范围清单稳定
- 页面和模块边界稳定
- 关键数据结构稳定
- 关键表结构和关键字段约束稳定
- 核心接口契约稳定
- 权限矩阵稳定
- 关键副作用规则稳定
- 验收标准可执行
- 任务拆解可排期
- `plan.md` 已给出自动开发就绪结论
- `analysis.md` 已给出固定格式的最终结论和缺口分类
- API、表、权限、副作用、验收文档已达到最小结构化规格

## 5. 两层之间的交接协议

第一层交给第二层的，不应是散乱文档，而应是一个业务产物包。

这个包至少必须回答下面六个问题：

1. 目标是什么
2. 谁参与
3. 流程怎么走
4. 规则怎么判断
5. 页面怎么分
6. 哪些地方还不能拍板

除此之外，若要支撑自动开发，还必须额外回答：

7. 哪些页面对应哪些用例
8. 哪些用例对应哪些接口和副作用
9. 哪些对象最终落到哪些表和字段
10. 哪些角色拥有哪些数据和动作权限

第二层接手后，要把这六类信息逐一映射为交付对象：

| 第一层结果 | 第二层映射 |
| --- | --- |
| 业务目标 | PRD 范围与目标 |
| 参与角色 | 权限与交互主体 |
| 流程图 | 页面流转与接口时序 |
| 业务规则 | 校验规则、写入规则、覆盖规则 |
| 状态图 | 状态管理与数据生命周期 |
| 验收样例 | 测试用例与验收清单 |
| 页面清单 | 页面到交付对象映射 |
| 核心对象 | 实体目录、表目录与字段规格 |

## 6.1 面向中型项目的规模化拆分规则

当 feature 规模接近中型项目时，应遵守以下规则：

- 不把所有流程塞进一张主图，主图只保留一级骨架
- 每条关键流程允许拆为独立 `FLOW-*` 文件
- 每个界面必须使用稳定 `SCREEN-*` 编号
- 界面文档允许按模块拆目录，不限于顺序编号命名
- 每个界面文档必须单独写明页面状态、显隐条件、可编辑条件和动作回链
- 每张数据库表必须单独建 `TABLE-*` 文档
- `API-*` 应按业务域或模块分组，而不是全部堆在一个章节中
- `plan.md` 只做总索引与范围说明，不承担所有细节正文

## 7. 当前阶段建议

基于当前仓库现状，下一步最合理的是先把第一层产物模板继续打磨完整，再为第二层补齐模板，而不是默认承接某一种实现框架。

当前优先级建议如下：

1. 固定第一层业务产物包结构
2. 固定第一层到第二层的交接协议
3. 为第二层补齐数据库、接口、权限和非功能模板
4. 固定跨文档编号和追踪规则
5. 再考虑具体项目上的技术实现方法
