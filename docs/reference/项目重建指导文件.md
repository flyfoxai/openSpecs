# 项目重建指导文件

本文用于未来基于新版 upstream `github/spec-kit` 重建 SP 项目骨架。

目标不是从旧 SP 项目里继续修补，而是每次遇到新版 upstream 时，先拿干净 upstream 做“瓶子”，再把 SP 已验证的内容增强迁进去。这样可以最大程度复用原版稳定的安装、集成、打包和宿主接入机制，减少这次迁移中出现的机制争议。

## 一句话原则

机制跟 upstream，内容保留 SP。

- CLI、integration registry、shared infrastructure、manifest、extension、preset、workflow、打包方式，默认按新版 upstream 原样保留。
- SP 的分层命令、memory、workset、flow、UI、delivery、gate、bundle、analyze 增强，作为内容层和少量 upstream-compatible hook 迁入。
- 如果某个改动会重写 upstream 主安装链，必须先证明 upstream 机制不能承载该需求；否则不要另起一套安装器。

## 重建总流程

1. 拉取干净 upstream 新版本，单独放在 `spec-kit-vX.Y.Z-upstream`。
2. 复制 upstream 目录作为新的 `speckit-layered-vX.Y.Z-sp` 工作树。
3. 把上一版 SP 成品整体归档到新工作树的 archive 目录，不要删除业务内容。
4. 先对比新版 upstream 与旧 upstream 的安装、integration、shared infrastructure、workflow、preset、packaging 变化。
5. 只在新版 upstream 仍然支持的机制位置迁入 SP 改动。
6. 迁入命令内容、项目 scaffold、memory、feature 模板树和文档。
7. 更新测试期望，不用全局替换 `speckit`。
8. 跑 source-tree smoke、integration tests、wheel smoke、真实宿主回归。
9. 把本文件按新版结果更新，记录新版本的实际操作差异。

- upstream 新增的安全检查、manifest 行为、集成选项、打包路径，优先保留。
- SP 只补“命名、内容、scaffold、memory、少量安装钩子”。

## 特性 1：内置命令改为 `sp.*` / `sp-*`

### 作用

这个特性解决的是入口命名问题。SP 已经不是原版 Spec Kit 的原样内容，而是在原版机制上增加了分层设计、memory、flow、UI、delivery 等能力，所以内置命令需要独立且稳定的业务入口。

### 保留规则

- dot 宿主优先使用 `sp.*`，例如 `/sp.plan`、`/sp.analyze`。
- skills 宿主使用 `sp-*`，例如 `/sp-plan`、`/sp-analyze`，目录为 `sp-plan/SKILL.md`。
- `sp.*` 是命令身份、阶段名、文档概念名。
- `sp-*` 是 skills 宿主调用和目录形式。
- extension、preset、community ecosystem 继续保留 upstream 的 `speckit.*` / `speckit-*`，不要一起改成 `sp`。

### 重建操作

1. 新增或迁移 `src/specify_cli/command_names.py`。
2. 在其中集中维护内置命令词干集合，例如 `specify`、`plan`、`tasks`、`analyze`、`flow`、`ui`、`gate`、`bundle`。
3. 让 `command_file_name()` 对内置命令返回 `sp.<stem>`。
4. 让 `command_invocation()` 对内置命令按 separator 返回 `/sp.<stem>` 或 `/sp-<stem>`。
5. 让 `skill_directory_name()` 对内置命令返回 `sp-<stem>`。
6. 修改 `src/specify_cli/integrations/base.py`，让 Markdown、TOML、YAML、Skills 生成路径统一调用这些 helper。
7. 修改 Copilot、Claude、Kimi、Forge、preset、extension 中所有手写 skill 目录或命令名的位置，让它们通过 helper，而不是各自拼字符串。
8. 增加或更新测试：`tests/integrations/test_base.py`、`tests/integrations/test_integration_base_skills.py`、`tests/integrations/test_integration_subcommand.py`、`tests/integrations/sp_expected.py`。

### 验收

- Markdown/TOML/YAML/Copilot 类宿主生成 `sp.plan.*`、`sp.analyze.*`。
- Codex、Claude、Kimi、Agy、Trae、Vibe、Devin、Lingma 等 skills 宿主生成 `sp-plan/SKILL.md`。
- extension 命令仍能生成或解析 `speckit.git.commit` / `speckit-git-commit`。
- 测试中不能把所有 `speckit.*` 全局替换为 `sp.*`。

## 特性 2：`__SPECKIT_COMMAND_*__` 可执行入口占位符

### 作用

这个特性解决的是不同宿主调用语法不同的问题。同一个“建议下一步执行 plan”在 dot 宿主里应该是 `/sp.plan`，在 skills 宿主里应该是 `/sp-plan`。如果模板里硬编码任意一种，就会让另一类宿主出错。

### 保留规则

- 需要用户或模型执行命令的句子，使用 `__SPECKIT_COMMAND_PLAN__` 这类占位符。
- 命令标题、阶段名、职责描述保留 `sp.plan` 这类概念名。
- 不要把 `before sp.gate` 这类阶段边界改成 `before /sp-gate`。

### 重建操作

1. 保留或补回 `IntegrationBase.resolve_command_refs()`。
2. 让它调用 `command_invocation()`，按 integration 的 `invoke_separator` 渲染。
3. `IntegrationBase.invoke_separator` 默认为 `.`。
4. `SkillsIntegration.invoke_separator` 设为 `-`。
5. Copilot skills mode 如果存在，也必须在 skills mode 下使用 `-`。
6. `shared_infra.py` 安装 `.specify/templates` 和 `templates/project` 文本文件时，也要调用 `resolve_command_refs()`。
7. `agents.py` 或相关 AGENT_CONFIGS 生成处必须传播 integration 的 `invoke_separator`。

### 验收

- `templates/commands` 中不应硬编码 `/sp.` 或 `/sp-` 作为可执行入口。
- 生成到 skills 宿主的文件中不残留 `__SPECKIT_COMMAND_`。
- 生成到 skills 宿主的文件中可执行提示使用 `/sp-*`。
- 生成到 dot 宿主的文件中可执行提示使用 `/sp.*`。

## 特性 3：`scripts:` 按 upstream 作为安装 / 渲染元数据

### 作用

这个特性解决的是“误以为宿主会硬执行 frontmatter 脚本”的问题。原版机制的稳定点在于安装期把 `{SCRIPT}` 渲染成具体脚本入口，而不是要求每个宿主都理解并自动执行 `scripts:`。

### 保留规则

- 命令模板可以保留 upstream 风格的 `scripts:` frontmatter。
- `scripts:` 只表示安装 / 渲染期选择哪个平台脚本。
- 安装后的宿主文件不应依赖 raw `scripts:` 自动执行。
- 只有需要脚本入口的命令才必须在正文中使用 `{SCRIPT}`。

### 重建操作

1. 保留 `IntegrationBase.process_template()` 中提取 `scripts.<script_type>` 的逻辑。
2. 保留把 `{SCRIPT}` 替换为具体命令的逻辑。
3. 保留剥离 `scripts:` / `agent_scripts:` 的逻辑。
4. 命令模板仍使用 upstream frontmatter 结构，不回到 agent 专用静态 prompt 目录。
5. 脚本输出必须清楚区分没有 active feature、文件缺失、目录不存在、scaffold 未填等状态。

### 验收

- 安装后的 `SKILL.md`、`.agent.md`、`.md`、`.toml`、`.yaml` 不残留未处理的 `{SCRIPT}`。
- 安装后的宿主文件不残留会误导宿主的 raw `scripts:`。
- `sp.analyze`、`sp.plan`、`sp.tasks` 等脚本型命令能通过正文脚本入口工作。

## 特性 4：新增分层命令 `flow`、`ui`、`gate`、`bundle`

### 作用

这个特性解决的是 upstream 默认 SDD 链路粒度偏粗的问题。原版有 specification、planning、tasks、implementation，但没有把业务流、页面表达、阶段闸门、跨层摘要作为显式步骤。SP 增加这些命令，让模型每次只处理一个清楚层面。

### 保留内容

- `sp.flow`：生成或刷新业务流程、分支、异常、状态、时序。
- `sp.ui`：生成或刷新页面地图、页面职责、动作、字段、JSON Forms 示例。
- `sp.gate`：记录阶段 readiness，防止未达标文档进入下一阶段。
- `sp.bundle`：把第一层稳定结论压缩成第二层可接手摘要。

### 重建操作

1. 把上一版 SP 的 `templates/commands/flow.md`、`ui.md`、`gate.md`、`bundle.md` 迁入新版 `templates/commands/`。
2. 把这四个词干加入 `CORE_COMMAND_STEMS` 或新版等价命令集合。
3. 更新 command template discovery / expected command tests。
4. 如果新版 upstream 增加命令注册 manifest，需要把这四个命令补进去。
5. 保留 frontmatter、handoffs、scripts、argument placeholder 的 upstream 格式，不改成旧版静态 prompt。

### 验收

- 所有主要 agent 安装后都能看到这四个命令。
- Codex/Claude skills 目录存在 `sp-flow`、`sp-ui`、`sp-gate`、`sp-bundle`。
- Copilot 或 dot 宿主存在 `sp.flow`、`sp.ui`、`sp.gate`、`sp.bundle`。

## 特性 5：两层文档推进模型

### 作用

这个特性解决的是业务未澄清就进入交付设计的问题。模型如果过早设计接口、表和任务，容易把假设当事实。两层模型先固化业务，再进入交付设计，降低返工。

### 保留内容

第一层业务澄清：

- 业务目标
- 角色与对象
- 主线流程
- 分支与异常
- 业务规则与状态
- 页面与交互表达
- 待确认问题与验收样例

第二层交付设计：

- PRD
- 页面到交付设计映射
- 用例矩阵
- 领域模型
- 数据实体
- 表规格
- API 契约
- 权限矩阵
- 事件与副作用
- 非功能需求
- 模块边界
- 测试与验收

### 重建操作

1. 把 `templates/project/.specify/templates/feature/flows/` 迁入新版。
2. 把 `templates/project/.specify/templates/feature/ui/` 迁入新版。
3. 把 `templates/project/.specify/templates/feature/delivery/` 迁入新版。
4. 检查 `templates/commands/clarify.md`、`flow.md`、`ui.md`、`gate.md`、`bundle.md`、`plan.md` 是否仍按两层边界读写文档。
5. 不把业务特定样例硬拷进通用模板；业务样例只作为质量参考。

### 验收

- 新项目初始化后存在 flow、UI、delivery 模板根。
- `sp.plan` 不只依赖 upstream 默认 `spec.md`，还知道读取 gate、bundle、delivery 和 memory。
- `sp.analyze` 会检查第一层和第二层之间是否可追踪。

## 特性 6：项目级 memory 与 feature 级 memory

### 作用

这个特性解决的是大模型反复全量读取、反复推理同一事实的问题。memory 不是替代源文档，而是让模型先定位、再展开，减少 token 消耗并降低上下文漂移。

### 保留内容

项目级 memory：

- `.specify/memory/project-index.md`
- `.specify/memory/active-context.md`
- `.specify/memory/constitution.md`
- `.specify/memory/feature-map.md`
- `.specify/memory/domain-map.md`
- `.specify/memory/hotspots.md`

feature 级 memory：

- `memory/index.md`
- `memory/stable-context.md`
- `memory/open-items.md`
- `memory/trace-index.md`
- `memory/worksets/index.md`
- `memory/worksets/ws-primary-journey.md`
- `memory/worksets/ws-decision-and-approval.md`
- `memory/worksets/ws-query-and-followup.md`
- `memory/worksets/ws-side-effects.md`

### 重建操作

1. 把项目级 memory 放入 `templates/project/.specify/memory/`。
2. 把 feature 级 memory 放入 `templates/project/.specify/templates/feature/memory/`。
3. 在 `shared_infra.py` 或新版等价 shared install 层中，增加 manifest-tracked project scaffold 安装步骤。
4. 安装 `templates/project/` 时对 `.md`、`.mmd`、`.json`、`.ps1`、`.sh` 调用 `resolve_command_refs()`。
5. 在 `pyproject.toml` wheel force-include 中包含 `templates/project`。
6. `sp.analyze`、`sp.plan`、`sp.tasks` 等命令必须要求先读 memory routing，再扩展到源文档。

### 验收

- `specify init` 后目标项目存在 `.specify/memory/project-index.md`。
- `specify init` 后目标项目存在 `.specify/templates/feature/memory/index.md`。
- memory 文件中的可执行命令提示经安装后能按宿主渲染。
- stale project memory 不能被无条件信任，feature-level memory 和当前源文档可以纠正它。

## 特性 7：workset 拆分

### 作用

这个特性解决的是大 feature 无法稳定放进单次上下文的问题。workset 把 feature 切成主旅程、审批决策、查询跟进、副作用等局部工作面，让模型每次只处理最小范围。

### 保留内容

- `ws-primary-journey.md`
- `ws-decision-and-approval.md`
- `ws-query-and-followup.md`
- `ws-side-effects.md`

### 重建操作

1. 迁入 `templates/project/.specify/templates/feature/memory/worksets/`。
2. 检查每个 workset 是否包含适用问题、纳入范围、不纳入范围、最小阅读集、完成判据。
3. 在命令模板中保留“先通过 workset 定位，再展开源文档”的规则。
4. 如果新版 upstream 有自己的上下文索引机制，优先让 workset 成为其内容输入，不另建运行引擎。

### 验收

- 新项目包含四个通用 workset 模板。
- 大 feature 的命令不会默认全量扫描所有 delivery、flow、ui 文件。
- workset 命中失败时，也只扩展最小源文档集。

## 特性 8：稳定 ID 与跨文档追踪

### 作用

这个特性解决的是自然语言引用不稳定的问题。页面、流程、接口、表、权限、验收路径如果只靠名字，模型容易混淆。稳定 ID 让 `sp.analyze` 能检查覆盖关系，也让后续自动化开发可以从任务回链到业务事实。

### 保留 ID 类型

- `ROLE-*`
- `OBJ-*`
- `RULE-*`
- `STATE-*`
- `STAGE-*`
- `FLOW-*`
- `STEP-*`
- `DEC-*`
- `EX-*`
- `SCREEN-*`
- `SECTION-*`
- `ACTION-*`
- `FIELD-*`
- `UC-*`
- `MOD-*`
- `API-*`
- `TABLE-*`
- `ACC-*`

### 重建操作

1. 迁入增强后的 `templates/spec-template.md`。
2. 迁入增强后的 `templates/plan-template.md`。
3. 迁入 `templates/project/.specify/templates/feature/memory/trace-index.md`。
4. 迁入 delivery 模板，并确认它们保留 ID 字段和回链字段。
5. 不在 `templates/project/.specify/templates/` 下再放重复的 `spec-template.md` 和 `plan-template.md`，新版 upstream 默认从顶层 `templates/` 安装它们。

### 验收

- 生成项目的 `.specify/templates/spec-template.md` 和 `plan-template.md` 包含 SP 的 ID / trace / memory 增强。
- `sp.analyze` 会检查 ID、owner、state、screen、API、table、permission、acceptance path 覆盖。

## 特性 9：feature 模板树增强

### 作用

这个特性解决的是新项目初始化后缺少标准 SP 文档落点的问题。如果没有模板树，模型会临时发明目录，或者漏掉 memory、flows、ui、delivery、gate、bundle、analysis 等文件。

### 保留内容

feature 根文件：

- `analysis.md`
- `bundle.md`
- `clarifications.md`
- `clarify-log.md`
- `gate.md`
- `tasks.md`

子目录：

- `memory/*`
- `flows/*`
- `ui/*`
- `ui/jsonforms/*`
- `delivery/*`
- `delivery/tables/*`

### 重建操作

1. 迁入 `templates/project/.specify/templates/feature/`。
2. 确保 shared infrastructure 安装时会复制整个 `templates/project/` 树。
3. 确保复制时受 manifest 跟踪。
4. 确保 install / upgrade / switch 不会覆盖用户改过的文件，除非 `--force` 或新版 upstream 等价参数明确要求。
5. 确保 symlink 安全检查沿用 upstream，不要为了复制目录绕过安全逻辑。

### 验收

- `specify init` 后出现 `.specify/templates/feature/analysis.md`。
- `specify init` 后出现 `.specify/templates/feature/delivery/01-prd.md`。
- manifest 中记录这些 scaffold 文件。
- `specify integration upgrade` 或新版等价命令不会无声覆盖用户改动。

## 特性 10：`sp.analyze` 增强为文档系统强度检查

### 作用

这个特性解决的是文档“看起来齐全，但不能支撑自动化”的问题。SP 的 analyze 不只看文件是否存在，还要检查 ID、owner、状态、页面、接口、表、权限和验收路径是否能连起来。

### 保留规则

- `sp.analyze` 可以创建或更新 `analysis.md`。
- `sp.analyze` 可以刷新相关 memory。
- 它只能做文档工作，不写生产代码。
- 如果 scaffold 占位仍未填，应输出 FAIL，不应误报 PASS。

### 重建操作

1. 迁入 `templates/commands/analyze.md`。
2. 迁入 `templates/project/.specify/templates/feature/analysis.md`。
3. 确认 `analyze.md` 先读 project memory，再读 feature memory，再按目标区域展开源文档。
4. 确认它检查 `spec.md`、`clarifications.md`、`flows/*`、`ui/*`、`gate.md`、`bundle.md`、`plan.md`、`delivery/*`、`tasks.md`。
5. 如果新版 upstream 的 analyze 机制变化，只保留其 frontmatter、scripts、argument、handoff 外壳，把 SP 的业务检查作为正文内容迁入。

### 验收

- 空 scaffold 项目执行 analyze 应 FAIL，并指出缺口。
- 已填完整文档执行 analyze 才能 PASS。
- analyze 生成或更新的 `analysis.md` 可回链到具体文档证据。

## 特性 11：安装机制保持 upstream 主链

### 作用

这个特性解决的是旧版自定义安装链不稳定的问题。之前按 agent 分目录静态拷贝，容易造成 Codex、Claude、Copilot 的格式、触发和安装落点漂移。现在必须回到 upstream `specify init`、integration registry、shared infra、manifest、packaging 主链。

### 保留规则

- 不恢复旧版 `installer-assets/claude-commands`、`codex-prompts` 作为主安装机制。
- 不为 Codex/Claude 单独维护另一套命令模板源。
- shared infrastructure 可以增加小的 project scaffold 安装钩子，但必须 manifest-tracked。
- 打包时必须保证 source install 和 wheel install 都能找到 `templates/project`。

### 重建操作

1. 保留新版 upstream `specify init` 主流程。
2. 在 shared infrastructure 层加入 `shared_project_source()` 或新版等价函数。

## 升级时最容易再次争议的边界

### `scripts:` 是否宿主硬执行

结论：不是。

按 upstream 做法，`scripts:` 是模板安装 / 渲染期元数据。它的作用是选择平台脚本并替换 `{SCRIPT}`。不要写成“所有宿主都会自动执行 frontmatter scripts”。

### `sp.*` 与 `sp-*`

结论：分层处理。

- `sp.*` 是 dot 宿主调用、命令身份和阶段概念。
- `sp-*` 是 skills 宿主调用和目录。
- 可执行提示使用 `__SPECKIT_COMMAND_*__`。
- 概念句保留 `sp.*`。

### `speckit.*` 是否全部替换

结论：不能全局替换。

内置命令迁到 `sp`。extension、preset、community ecosystem、workflow 标识、manifest 里属于 upstream 生态的名字继续保留 `speckit`。

### `analyze` 写文件是否机制冲突

结论：不是机制冲突，是 SP 内容增强。

只要安装外壳、frontmatter、script 渲染、宿主入口仍按 upstream，`sp.analyze` 允许写 `analysis.md` 和刷新 memory 属于业务行为增强。
