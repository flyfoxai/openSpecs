# `sp` 中型项目冒烟测试规范

> 目标：把“这套文档是否足以支撑中型项目后续自动开发”变成一份可执行检查，而不是口头判断。

## 1. 适用范围

本规范适用于接近以下规模的 feature 或模块：

- 约 `50` 个界面
- 约 `50` 条流程
- 约 `100` 张表

也适用于还没达到这个规模，但已经开始出现以下迹象的项目：

- 文档明显分卷
- 页面、流程、表已经跨多个业务域
- 单人已无法靠记忆维护追踪关系

## 2. 使用方式

这份规范不引入新命令。

建议用法：

1. 先完成 `sp.specify` 到 `sp.tasks`
2. 在 `sp.analyze` 中执行本页检查
3. 将结果写入 `analysis.md`
4. 再给出 `PASS`、`PASS WITH RISKS` 或 `BLOCKED`

如果是中型项目或接近中型项目，`analysis.md` 不应只写“总体一致”。

必须显式写出：

- 本次冒烟范围
- 抽样范围
- 通过项
- 风险项
- 阻塞项
- 建议回退步骤

## 3. 冒烟检查维度

### 3.1 规模拆分检查

必须确认：

- flow 已按业务域或子主题拆分
- ui 已按模块或场景拆分
- API 契约不是全部堆在一个自然语言章节里
- 表规格已做到逐表成文
- 至少存在一个索引文件维护编号、名称和关系
- `plan.md` 只承担总索引和范围说明，不吞掉全部细节

任一项缺失，至少记一条 `Medium` 风险。

如果流程、页面、表仍主要依赖单文件堆叠，应直接记为结构性缺口。

### 3.2 编号与追踪检查

必须确认关键对象具备稳定编号：

- `FLOW-*`
- `SCREEN-*`
- `UC-*`
- `API-*`
- `TABLE-*`
- `ACC-*`

并且必须能从文档中显式回链：

- `SCREEN -> FLOW -> RULE`
- `SCREEN -> UC -> API`
- `API -> OBJ -> TABLE`
- `TASK -> SCREEN / API / TABLE / ACC`
- `ACC -> SCREEN / API / TABLE / RULE`

如果这些关系仍然需要靠人工猜，应直接记为 `High` 风险。

### 3.3 UI 精度检查

至少抽查关键页面是否写明：

- 页面状态
- 区块显隐条件
- 字段可编辑条件
- 提交前校验
- 提交后反馈
- 对应 `UC-*`
- 必要时对应 `API-*`

如果关键页面只有页面简介，没有状态与动作约束，应记为 `UI` 类缺口。

### 3.4 API 精度检查

至少抽查关键接口是否写明：

- 请求字段
- 响应字段
- 字段类型
- 必填与可空
- 默认值或枚举
- 错误码
- 权限前提
- 状态前提
- 幂等要求

如果接口文档只有“做什么”，没有字段级契约，应记为 `API` 类缺口。

### 3.5 数据精度检查

至少抽查关键表是否写明：

- 字段类型与长度
- 必填或可空
- 默认值
- 主键、唯一约束、索引
- 外键或关联
- 删除策略
- 审计字段
- 状态字段映射

如果表文档只有表名和用途，没有字段级规则，应记为 `Data` 类缺口。

### 3.6 权限与副作用检查

至少抽查：

- 关键角色是否有页面显隐规则
- 关键动作是否有字段编辑权限
- 是否有数据范围限制
- 审批、通知、审计、余额变更等副作用是否单独成文
- 副作用是否标明同步异步、幂等键、失败处理

如果权限和副作用仍混在自然语言说明里，应分别记为 `Permission` 或 `Side-Effect` 类缺口。

### 3.7 验收检查

至少抽查关键验收项是否写明：

- 前置数据
- 操作步骤
- 预期结果
- 异常结果
- 回链到 `SCREEN-* / API-* / TABLE-* / RULE-*`

如果验收项无法回链页面、接口、表和规则，应记为 `Acceptance` 类缺口。

## 4. 抽样规则

中型项目不要求逐项全量重读，但不能只挑一个页面做样子。

建议最低抽样如下：

- `flows`
  - 全量检查关键主流程和高风险流程
  - 其余至少抽样 `20%`，且不少于 `10` 条
- `screens`
  - 全量检查关键入口页、审批页、配置页、详情页
  - 其余至少抽样 `20%`，且不少于 `10` 个
- `tables`
  - 全量检查核心交易表、审批表、状态表、审计表
  - 其余至少抽样 `15%`，且不少于 `15` 张
- `apis`
  - 全量检查写操作接口和外部集成接口
  - 其余至少抽样 `20%`
- `acceptance`
  - 全量检查关键主链路验收项
  - 其余至少抽样 `20%`

关键对象必须优先进入样本：

- 审批链路
- 涉及删除或撤回
- 涉及金额、余额、额度
- 涉及组织范围、数据权限
- 涉及外部通知、审计、异步任务

## 5. 判定规则

### 5.1 可判 `PASS`

只有同时满足以下条件，才可判 `PASS`：

- 规模拆分规则已成立
- 关键追踪关系可显式回链
- 抽样中未发现 `High` 缺口
- 即使存在问题，也只是局部补充，不影响自动开发边界判断
- `analysis.md` 已明确记录冒烟范围、抽样范围和结论依据

### 5.2 可判 `PASS WITH RISKS`

出现以下情况，可判 `PASS WITH RISKS`：

- 主链路已完整，但部分页面、接口、表的精度还不均匀
- 结构已成型，但局部编号或回链仍需补齐
- 关键权限或副作用已经成文，但少数边角场景还不够细

此时必须在 `analysis.md` 中写明：

- 风险项编号
- 影响范围
- 建议回退步骤
- 是否允许进入受控自动开发

### 5.3 必须判 `BLOCKED`

出现以下任一情况，应直接 `BLOCKED`：

- `SCREEN -> UC -> API -> TABLE` 主链路无法稳定回链
- 审批、删除、外部通知等高风险链路没有副作用定义
- 关键页面没有状态、显隐、可编辑规则
- 关键接口没有字段级契约
- 关键表没有字段级规格
- 验收项无法证明主链路完成条件
- 文档仍主要依赖单文件堆叠，无法支持中型项目拆分

## 6. 考勤请假申请样例

以“请假申请”作为样例，冒烟时至少应抽到：

- `SCREEN-LEAVE-APPLY`
- `SCREEN-LEAVE-APPROVAL`
- `UC-LEAVE-SUBMIT`
- `UC-LEAVE-APPROVE`
- `API-LEAVE-CREATE`
- `API-LEAVE-APPROVE`
- `TABLE-LEAVE_REQUEST`
- `TABLE-LEAVE_APPROVAL`
- `ACC-LEAVE-APPROVE-SUCCESS`

至少应能验证：

- 不同假种的字段显隐和可编辑条件
- 提交前余额校验规则
- 审批通过后的余额占用、通知、审计
- 驳回或撤回时的状态回退与副作用处理
- 审批页中不同角色的可见字段差异
- 验收项与页面、接口、表、规则都能回链

如果这些点中有两项以上仍然只能靠口头推断，不应判 `PASS`。

## 7. 推荐写入 `analysis.md` 的结构

若 feature 达到中型项目规模，建议在 `analysis.md` 额外增加：

```md
## Medium-Project Smoke Test

### Scope

### Sampling Summary

| Domain | Total | Sampled | Critical Included | Result |
| --- | --- | --- | --- | --- |

### Findings

### Verdict Impact
```

其中：

- `Scope` 写本次判定覆盖的业务域或模块
- `Sampling Summary` 写抽样数量和关键对象是否纳入
- `Verdict Impact` 写这轮冒烟如何影响最终 `PASS / PASS WITH RISKS / BLOCKED`

## 8. 与现有文档的关系

本规范不是替代以下文档，而是把它们落成检查动作：

- [docs/sp-auto-development-readiness.md](sp-auto-development-readiness.md)
- [docs/sp-auto-development-minimum-spec.md](sp-auto-development-minimum-spec.md)
- [docs/sp-auto-development-passline.md](sp-auto-development-passline.md)

最简理解：

- `readiness` 负责判断方向是否正确
- `minimum-spec` 负责判断还缺哪些精度
- `passline` 负责定义通过线
- 本页负责告诉你如何在 `sp.analyze` 里做一次像样的中型项目冒烟检查
